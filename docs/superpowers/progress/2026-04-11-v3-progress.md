### Critical discovery during testing

The verify script's browser check showed the v3 `<video>` element at `readyState=0` throughout the smoke test. I attributed this to "headless chromium MSE codec limitation" in the self-audit. That was wrong. The actual cause was **go2rtc returning 403 on the MSE WebSocket handshake because the dashboard origin (127.0.0.1:8099 or 8199) didn't match go2rtc's origin (127.0.0.1:1984)**.

This means David's "framerate is still low" complaint was NOT that v3 is slow — it's that **v3's MSE stream never connected, so the video element was always empty**. Whatever video he was seeing was the OLD v2 live-video element, which uses MJPEG from the server at YOLO rate (1-3 fps).

Fix: route the WebSocket through `dashboard/api.py`'s existing `proxy_go2rtc_ws` route at `/api/ws` (same origin as the dashboard page). Browser → uvicorn → go2rtc. No 403.

This should finally unlock the "smooth 30fps HD" promise of the v3 architecture. Awaits David's real-browser verification.

### Sync-delay implementation (the thing he asked for)

Video pauses for `SYNC_DELAY_MS` (default 3000ms, localStorage-tunable) on first `canplay` → video plays ~3s behind wall clock. SSE events go into a `pendingEvents` queue keyed by server `wall_time_ms`. Render loop drains events where `wall_time_ms <= Date.now() - SYNC_DELAY_MS` and applies them to `trackStates`. All track timing (`first_seen_t`, `prev_t`, `last_t`, `fadeOutAt`, fade-in/out, interpolation) now uses server wall clock instead of browser monotonic time.

Expected UX: when a bird walks into the visible frame (from the delayed video), its label has been in the browser's buffer for ~2500ms and is already positioned. Labels lead the bird into frame, don't follow it.
